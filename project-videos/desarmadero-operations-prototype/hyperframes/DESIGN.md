# Design

## Style Prompt

Use an operational case-study style: grounded, structured, and practical. The video should feel like one discovery call being translated into a working yard operating model, not like a generic SaaS promo. Use the two public prototype screenshots as proof anchors, then surround them with clear cards, rails, role labels, and map callouts. The copy stays simple and nontechnical: problem, cost, prototype, workflow, result, and scope clarity.

## Concept Angle

A messy yard operation is turned into a visible operating map: paper, messages, spreadsheets, and memory become role surfaces, handoff rails, and searchable yard state.

## Typography

- Display: `League Gothic`, condensed display
- Interface/body: `Montserrat`, sans-serif
- Data labels: `IBM Plex Mono`, monospace

## Motion

- Six scenes, `45s` total.
- Use quiet but visible motion: cards assemble, rails reveal, screenshots settle, map callouts lock in.
- Use push slides for forward story beats, a blur crossfade into the map result, and a gentle push into the final value scene.
- Add ambient movement to background panels, grid, and map texture so scenes stay alive without distracting from readable text.
- Keep product-screenshot scenes readable for the full hold.
- The final `2.5-3s` must be a settled ending beat with the project name and one short result line.

## What Not To Do

- Do not show credentials, passwords, transcript contents, private client details, local paths, account state, tokens, or raw data.
- Do not put framework names, database names, API terms, CLI commands, or test counts on screen.
- Do not make the video only title cards; the prototype screenshots must carry proof.
- Do not use neon, purple-blue gradients, or dark generic SaaS styling.
- Do not use jump cuts between scenes.
