<CaseSection title="Summary">
	Job Application Assistant is a local workflow tool for drafting and tracking job applications.

    It combines a Chrome extension, a Python FastAPI backend, a SQLite ledger, local portfolio context, Codex draft generation, and cover-letter PDF export into one auditable loop.

</CaseSection>

<CaseSection title="The Situation">
	The job application workflow had too many disconnected states.

    A job board had the source posting. Local notes had positioning and proof points. A draft lived in a chat or text box. A cover-letter PDF had to be generated separately. The application history lived somewhere else. That made it easy to lose the link between the role, the chosen proof, the final language, and whether the application was already sent.

</CaseSection>

<CaseSection title="Operating Model">
	<DecisionBlock title="Keep sensitive workflow state local.">
		The tool stores extension state in Chrome, draft jobs and application records in SQLite, generated PDFs under the configured runtime directory, and generated context under a local cache. The README is explicit that drafting uses the locally configured Codex/OpenAI account, and the app does not include analytics or job-board API calls.
	</DecisionBlock>

    <DecisionBlock title="Make extraction a contract.">
    	Supported job pages are normalized into an OpportunitySnapshot with fields like source URL, title, company, location, description, responsibilities, requirements, skills, questions, company context, recruiter context, and extraction warnings. Missing data stays missing with warnings instead of being filled from broad page text.
    </DecisionBlock>

    <DecisionBlock title="Make drafting traceable.">
    	The backend builds a single prompt from the job snapshot plus local profile, resume, offers, and project proof points. The model must return strict JSON with the draft and the audit fields needed to inspect why an angle or claim was used.
    </DecisionBlock>

</CaseSection>

<CaseSection title="The Work">
	<DecisionBlock title="Built the extension and extraction layer.">
		The extension supports Upwork, Dice, Indeed, ZipRecruiter, Robert Half, and LinkedIn page shapes. It uses platform adapters, structured selectors, Dice JobPosting JSON-LD where available, and reviewable side-panel fields before a draft job starts.
	</DecisionBlock>

    <DecisionBlock title="Built the local context and drafting backend.">
    	The backend indexes profile Markdown, offer JSON, project proof JSON, and resume text into a ContextBundle. Draft jobs run asynchronously, persist request and result JSON, expose polling status, and record stage and Codex subprocess timing metadata.
    </DecisionBlock>

    <DecisionBlock title="Added application tracking as part of the workflow.">
    	Application records are stored in SQLite and deduplicated by normalized source URL. The extension can check whether the current source URL already exists, and the local dashboard supports search, filters, date views, sorting, pagination, and draft links.
    </DecisionBlock>

    <DecisionBlock title="Turned cover letters into generated artifacts.">
    	The PDF exporter renders the saved draft text without rewriting it, derives a restrained letterhead from the configured resume PDF, omits phone-like contact items, names files by company, reveals generated files in Finder, and archives submitted cover letters when an application is logged.
    </DecisionBlock>

    <DecisionBlock title="Bounded apply automation to supported flows.">
    	The result-page side panel can list apply-enabled postings and open selected Dice, Indeed, and LinkedIn flows. Platform helpers can attach generated cover-letter PDFs and record confirmed applications, while unfamiliar application questions require manual attention.
    </DecisionBlock>

</CaseSection>

<CaseSection title="Why It Matters">
	The useful part is not that the system can write a cover letter.

    The useful part is that every draft is tied back to the job snapshot and the allowed portfolio proof, every application can be checked against the ledger, and every generated PDF remains part of the same local workflow.

</CaseSection>
