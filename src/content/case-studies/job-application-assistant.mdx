<CaseSection title="Summary">
	Job Application Assistant is a local tool for drafting and tracking job applications.

    It combines a Chrome extension, a FastAPI backend, SQLite, local portfolio context, Codex draft generation, and cover-letter PDF export. The important part is that the draft, the source job, the chosen proof, and the application ledger stay connected.

</CaseSection>

<CaseSection title="The situation">
	My application workflow had too many loose pieces.

    The job board had the posting. Local files had my positioning and project proof. Drafts lived in a chat or text box. Cover letters had to become PDFs somewhere else. Application history was another habit to maintain. It was too easy to lose the link between the role, the claim I made, the final PDF, and whether I had already applied.

</CaseSection>

<CaseSection title="How it works">
	<DecisionBlock title="Keep sensitive state local.">
		The extension keeps browser state in Chrome. The backend stores draft jobs and application records in SQLite. Generated PDFs live under the configured runtime directory, and generated context stays in a local cache. The README is explicit that drafting uses the locally configured Codex/OpenAI account. The app does not add analytics or job-board API calls.
	</DecisionBlock>

    <DecisionBlock title="Treat extraction as a contract.">
	Supported job pages become an OpportunitySnapshot with fields like source URL, title, company, location, description, responsibilities, requirements, skills, questions, company context, recruiter context, and extraction warnings. Missing data stays missing.
    </DecisionBlock>

    <DecisionBlock title="Make the draft inspectable.">
	The backend builds one prompt from the job snapshot plus local profile, resume, offers, and project proof. The model has to return strict JSON, including the draft and the audit fields needed to check why a claim was used.
    </DecisionBlock>

</CaseSection>

<CaseSection title="What I built">
	<DecisionBlock title="Extension adapters for real job pages.">
		The extension supports Upwork, Dice, Indeed, ZipRecruiter, Robert Half, and LinkedIn page shapes. It uses platform adapters, structured selectors, Dice JobPosting JSON-LD when available, and reviewable side-panel fields before a draft starts.
	</DecisionBlock>

    <DecisionBlock title="A local context and drafting backend.">
	The backend indexes profile Markdown, offer JSON, project proof JSON, and resume text into a ContextBundle. Draft jobs run asynchronously, persist request and result JSON, expose polling status, and record stage and Codex subprocess timing metadata.
    </DecisionBlock>

    <DecisionBlock title="Application tracking in the same workflow.">
	Application records are stored in SQLite and deduplicated by normalized source URL. The extension can check whether the current source URL already exists, and the local dashboard supports search, filters, date views, sorting, pagination, and draft links.
    </DecisionBlock>

    <DecisionBlock title="Cover-letter PDFs as saved artifacts.">
	The PDF exporter renders the saved draft text without rewriting it, derives a restrained letterhead from the configured resume PDF, omits phone-like contact items, names files by company, reveals generated files in Finder, and archives submitted cover letters when an application is logged.
    </DecisionBlock>

    <DecisionBlock title="Apply automation with review points.">
	The result-page side panel can list apply-enabled postings and open selected Dice, Indeed, and LinkedIn flows. Platform helpers can attach generated cover-letter PDFs and record confirmed applications. Unfamiliar application questions stay manual.
    </DecisionBlock>

</CaseSection>

<CaseSection title="Why it matters">
	The useful part is not that the system can write a cover letter.

    The useful part is that every draft stays tied to the job snapshot and the allowed portfolio proof. Every application can be checked against the ledger, and every generated PDF remains part of the same local workflow.

</CaseSection>
