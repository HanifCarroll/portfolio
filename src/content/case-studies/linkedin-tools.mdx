<CaseSection title="Summary">
	LinkedIn Tools is a public Python monorepo for guarded LinkedIn workflow automation.

    It brings networking, recruiter/agency outreach, opportunity intelligence, comment extraction, browser artifacts, and review surfaces under one controller-led system instead of scattered browser scripts.

</CaseSection>

<CaseSection title="The Situation">
	Browser automation is easy to make impressive and hard to make trustworthy.

    The risky part was not clicking LinkedIn buttons. It was knowing which workflow owned each action, whether the action was allowed, what evidence was captured, and how to reconcile the result when the browser or platform state was uncertain.

</CaseSection>

<CaseSection title="Operating Model">
	<DecisionBlock title="Make workflow ownership explicit.">
		Connection requests, acceptance tracking, accepted follow-ups, and pending-invitation cleanup live under the network controller. Recruiter/agency outreach owns account sourcing, lead capture, drafts, dashboards, dry-runs, and guarded message sends. Opportunity intelligence ranks and reviews buyer-signal comments but does not send, connect, or withdraw.
	</DecisionBlock>

    <DecisionBlock title="Keep real actions close to approval flags.">
    	Send-capable commands keep real-action flags next to the browser operation. The code distinguishes guarded dry-runs from explicit allow-send and allow-withdraw paths instead of letting a background script decide.
    </DecisionBlock>

    <DecisionBlock title="Treat audit evidence as product surface.">
    	The system stores source runs, capture artifacts, audit results, candidate reservoirs, acceptance ledgers, extracted comments, ranked comments, browser artifacts, and blocked states so each run can be inspected after the fact.
    </DecisionBlock>

</CaseSection>

<CaseSection title="Product Surface" media>
	The local review UI is a FastAPI and Jinja surface for opportunities, network state, recruiter/agency queues, browser artifacts, and token-gated guarded actions.

    The screenshot below uses isolated temporary state so the portfolio can show the shipped UI without exposing local LinkedIn runtime data.

</CaseSection>

<CaseSection title="The Work">
	<DecisionBlock title="Built the network automation controller.">
		The network namespace owns Sales Navigator connection-request runs, saved-search capture, candidate reservoirs, sent-page audit reconciliation, acceptance checks, accepted follow-up drafts, browser sessions, and pending-invitation cleanup.
	</DecisionBlock>

    <DecisionBlock title="Separated recruiter and agency outreach.">
    	The recruiter/agency namespace handles account sourcing, lead capture, message drafting, dashboarding, dry-runs, and guarded message sends. Tests enforce key boundaries such as requiring Sales Navigator identity before promoted agency contacts can become send-ready leads.
    </DecisionBlock>

    <DecisionBlock title="Kept opportunity intelligence recommend-only.">
    	The opportunity namespace defines source registries, query packs, provider imports, post queues, browser-backed search capture, source experiments, and review exports. The v0 source batch includes 38 enabled sources and a scoring model for problem fit, buying signal, buyer fit, actionability, and immediacy.
    </DecisionBlock>

    <DecisionBlock title="Made comment extraction resumable and inspectable.">
    	The comments namespace supports saved-HTML and browser-backed LinkedIn post comment extraction, safety limits, progress output, SQLite persistence, artifacts, queue checkpoints, and retry metadata.
    </DecisionBlock>

    <DecisionBlock title="Added a local review UI over real state contracts.">
    	The review UI reads from SQLite and JSON state to show opportunity sources, post queues, extraction runs, ranked comments, experiments, calibration rows, network runs, acceptance drafts, pending cleanup, recruiter leads, browser sessions, and guarded actions.
    </DecisionBlock>

</CaseSection>

<CaseSection title="Why It Matters">
	This is the same operating pattern I would use inside a company when a high-value workflow has outgrown spreadsheets, chat threads, and ad hoc scripts.

    Define ownership. Store the workflow state. Gate irreversible actions. Capture evidence. Make the next action visible. Let automation run only where the system can explain what happened.

</CaseSection>
